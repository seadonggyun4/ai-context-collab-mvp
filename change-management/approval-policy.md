# Human Approval Policy

## 문서 메타데이터

| 항목 | 값 |
| --- | --- |
| status | `ACTIVE` |
| scope | 변경 위험 분류와 사용자 승인 게이트 |
| approved_by | 프로젝트 사용자 |
| effective_at | `2026-07-16` |
| supersedes | 없음 |

## 목적

단일 AI의 Self-Review는 독립 승인이 아니다. 고위험 또는 해석 불확실성이 큰 변경에서는 사용자가 검증자이자 승인자 역할을 수행한다.

## 위험 수준

| 위험도 | 기준 | AI가 승인 전 할 수 있는 일 | 승인 전 금지 |
| --- | --- | --- | --- |
| 낮음 | 되돌리기 쉽고 권한·데이터·운영 영향이 없는 제한적 변경 | Context 확인, 계획, 구현, 검증 | 알려진 blocker를 숨긴 완료 선언 |
| 중간 | 여러 파일·역할에 영향이 있으나 안전하게 되돌릴 수 있음 | 분석, 계획, 초안, 로컬 검증 | 외부 적용이나 되돌리기 어려운 작업 |
| 높음 | 아래 필수 승인 대상에 해당 | read-only 조사, 영향 분석, 초안 제안 | 실제 적용, 삭제, migration, 배포, 공개 계약 확정 |

## 사용자 승인 필수 대상

- 인증과 권한
- 민감정보 수집·저장·전송·노출
- 데이터 삭제 또는 마이그레이션
- 외부 공개 인터페이스나 호환성 계약 변경
- 배포 및 운영 설정
- 되돌리기 어려운 변경
- 활성 기준 문서 충돌
- 요구사항을 여러 방식으로 해석할 수 있고 선택에 따라 결과가 크게 달라지는 변경

위 목록에 없더라도 실패 영향이 크거나 사용자 의사결정 없이는 안전한 기본값을 선택할 수 없으면 높음으로 분류한다.

## 승인 절차

1. Planner가 위험도와 근거를 Manifest에 기록한다.
2. 높음이면 구현 전 `approval-record-template.md`로 승인 요청 초안을 작성한다.
3. AI는 영향 분석과 비파괴적 초안까지만 진행하고 실제 적용을 멈춘다.
4. 사용자 응답의 승인 범위, 조건, 제외 사항을 Approval Record에 기록한다.
5. 승인 범위 밖 변경이 필요해지면 기존 승인을 확장 해석하지 않고 다시 요청한다.
6. 대상·범위·위험 조건이 바뀌면 기존 승인은 무효화하고 새 승인을 받는다.

사용자의 명시적 요청 자체가 구체적인 외부 작업을 포함하는 경우 그 작업 범위에 대한 승인으로 기록할 수 있다. 다만 요청에 포함되지 않은 파괴적 하위 작업까지 승인된 것으로 확대하지 않는다.

## 승인 상태

| 상태 | 의미 |
| --- | --- |
| `NOT_REQUIRED` | 낮음·중간 위험이며 별도 승인 불필요 |
| `PENDING` | 사용자 결정 대기, 실제 적용 금지 |
| `APPROVED` | 기록된 범위와 조건 안에서만 진행 가능 |
| `REJECTED` | 해당 접근으로 진행 금지 |
| `EXPIRED` | 범위·대상·조건 변경으로 재승인 필요 |

## 완료 조건

고위험 변경은 Approval Record가 `APPROVED`가 아니면 `DEVELOPMENT_READY`로 전환할 수 없다. 승인 없이 만든 초안은 완료 구현이 아니며 `DRAFT_ONLY`로 보고한다.
