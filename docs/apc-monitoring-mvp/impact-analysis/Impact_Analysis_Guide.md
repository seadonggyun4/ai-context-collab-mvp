# Impact Analysis Guide

## 목적

impact-analysis 문서는 기획, 퍼블리싱, 개발, QA 중 하나의 기준이 변경되었을 때 다른 직군 문서와 구현 범위에 어떤 영향이 있는지 기록하는 문서다.

AI는 변경 요청을 받을 때 이 문서를 기준으로 영향 범위를 정리하고, 필요한 문서 갱신 대상을 제안한다.

모든 Impact Analysis는 해당 Change Manifest에서 참조한다. Change Manifest에는 예상 영향 범위와 실제 변경 범위를 요약하고, 이 문서는 역할별 파급 효과와 후속 조치를 상세화한다. 두 문서가 다르면 구현 후 확인된 실제 변경을 기준으로 차이와 사유를 양쪽에 기록한다.

Impact Analysis는 기준 문서를 대체하지 않는 증거와 변경 이력이다. 현재 기준은 `../Active_Context.md`에 `ACTIVE`로 등록된 문서에서만 결정한다.

## 작성 시점

- Project Context의 목적, 사용자, 요구사항, 제약사항이 변경될 때
- Planning 메뉴/기능/상태/예외 정책이 변경될 때
- Publishing 디자인 시스템, 토큰, 컴포넌트 정책이 변경될 때
- Development API, 타입, 데이터 흐름, 배포 방식이 변경될 때
- QA 체크 항목 또는 수용 기준이 변경될 때

## 영향 분석 항목

| 항목 | 설명 |
| --- | --- |
| 변경 요약 | 무엇이 변경되었는가 |
| 변경 원인 | 왜 변경되었는가 |
| 영향 문서 | 수정이 필요한 문서 목록 |
| 기획 영향 | 사용자 흐름, 메뉴, 상태, 예외 정책 변경 여부 |
| 퍼블리싱 영향 | 컴포넌트, 토큰, 반응형, 접근성 변경 여부 |
| 개발 영향 | API, 타입, fixture, adapter, 배포 변경 여부 |
| QA 영향 | 테스트 케이스와 체크표 변경 여부 |
| 리스크 | 누락 시 발생할 수 있는 문제 |
| 후속 조치 | 문서 또는 구현 변경 TASK |
| Change ID | 연결된 `CR-YYYY-NNN`과 Manifest 경로 |
| 예상 대비 실제 | 계획한 파일·역할 영향과 실제 변경의 차이 |
| Active Context | 변경 전후 활성 기준 문서와 충돌 여부 |
| 승인 영향 | 위험도와 사용자 승인 필요 여부 |

## 파일명 규칙

```text
YYYY-MM-DD_change-summary.md
```

예시:

```text
2026-07-09_astryx-design-system-alignment.md
```
