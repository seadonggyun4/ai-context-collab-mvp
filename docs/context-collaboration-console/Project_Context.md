## 제품 정의

기획·퍼블리싱·개발·QA가 자연어로 변경을 요청하고, AI가 활성 Context와 조직 정책을 기준으로 영향 분석·역할별 산출물·검증 계획을 제안하며, 사람이 승인과 완료를 통제하는 협업 운영 콘솔이다.

이 제품은 AI 채팅 화면이 아니다. 프로젝트의 변경 요청, 기준 문서, 구현 범위, 승인, 테스트 증거와 활성 Context를 연결해 운영하는 control plane이다.

## 해결할 문제

- 기획 변경이 개발·QA 기준까지 전파되지 않는다.
- 문서와 실제 구현이 시간이 지나며 달라진다.
- AI 요청마다 서로 다른 문맥이 전달된다.
- 누가 어떤 근거로 승인했는지 추적하기 어렵다.
- 비개발자가 Markdown, YAML, Git을 직접 검수하기 어렵다.
- AI의 완료 선언과 실제 검증 증거가 분리된다.

## 사용자와 핵심 판단

| 사용자 | 콘솔에서 내려야 할 판단 |
| --- | --- |
| 기획자 | 요청 해석과 수용 기준이 업무 의도와 맞는가 |
| 퍼블리셔/디자이너 | 화면·컴포넌트·접근성 영향이 빠짐없는가 |
| 개발자 | API·타입·상태·파일·테스트 범위가 구현 가능한가 |
| QA | 수용 기준과 회귀 범위를 실제로 검증했는가 |
| 승인자/리더 | 위험·미검증·diff를 이해하고 적용을 승인할 수 있는가 |

## 핵심 기능

| ID | 기능 | 사용자 결과 |
| --- | --- | --- |
| CAP-01 | 프로젝트 대시보드 | 활성 Context, 진행 요청, 최신 산출물, 미승인·미검증, QA와 정합성을 한 화면에서 판단 |
| CAP-02 | 자연어 변경 요청 | 요청을 수용 기준·영향 역할·화면/API/데이터·예상 파일·위험·QA 시나리오로 구조화 |
| CAP-03 | 역할별 Context | Markdown을 역할별 읽기·검토 화면으로 제공하고 상위 기준과 변경 이력 표시 |
| CAP-04 | 변경 영향 그래프 | 요청에서 기준·계약·컴포넌트·구현·QA로 이어지는 관계 탐색 |
| CAP-05 | 승인과 검증 | before/after diff, AI 근거, 자동·수동 검증, 승인·반려·수정 요청, 완료 판정 |
| CAP-06 | 문서 원본 보기 | UI에서 편집한 결과의 Markdown/YAML 원본과 Git 상태 확인 |

## 핵심 시나리오

> 모니터링 목록에 ‘최근 정상 수신 시간’을 추가하고, 24시간 이상 수신되지 않으면 경고로 보여주세요.

1. 기획자가 자연어 요청을 등록한다.
2. 시스템은 요청을 변경 제안으로 저장하고 활성 Context snapshot을 고정한다.
3. AI는 수용 기준, 영향 역할, 화면·API·데이터, 파일, 위험, QA 시나리오를 제안한다.
4. 사용자는 역할별 변경안과 관계 그래프를 검토한다.
5. 승인자가 적용 범위를 승인하거나 반려·수정 요청한다.
6. 승인된 범위에 한해 구현·테스트 단계가 진행된다.
7. 자동 테스트와 수동 확인 결과가 분리돼 기록된다.
8. 완료 게이트 통과 후 새 Context가 활성화된다.

## 범위

### 최초 릴리스에 포함

- `apc-monitoring-mvp` 단일 프로젝트 fixture
- 실제 문서 구조를 반영한 읽기 중심 Context 브라우저
- 결정적인 demo scenario 기반 영향 분석
- 변경 요청 상태 전이, 승인·반려·수정 요청
- 역할별 산출물, semantic diff, 영향 그래프, QA evidence
- 반응형 웹과 키보드 접근성

### 후속 연동 경계

- 실제 LLM provider 호출, GitHub remote PR, 알림, 외부 CI 실행
- 다중 조직·다중 프로젝트 권한

### 제외

- 범용 문서 편집기
- 자유 형식 AI 채팅
- 승인 없는 자동 코드 변경·배포
- 소스 코드 IDE 대체

### 구현 범위 갱신

- 문서 원본 편집, revision 충돌 감지, schema 검증과 승인 제안 생성은 backend 연동 범위에 포함한다.
- 자유로운 repository 전체 코드 편집은 계속 제외한다. 편집 대상은 프로젝트 문서 정책으로 허용된 Markdown/YAML path다.
- 현재 릴리스는 login/session/account UI 없이 console에 직접 진입한다.
- 승인·검증·self-approval 차단은 fixture actor를 사용하는 업무 규칙으로 유지한다.
- 조직 OIDC, 실제 사용자 claim 기반 RBAC와 IdP application 등록은 별도 Change Manifest 승인 전까지 후속 범위다.
- Render production resource 생성과 PITR/rollback 실행은 운영 승인 후 runbook evidence로 완결한다.

## 저작자·소유권 경계

- 문서엔진 콘솔의 독창적 프론트엔드·백엔드 소스와 전용 문서의 저작자·권리자는 서동균(DongGyun Seo)으로 표시하고 모든 권리를 유보한다.
- 회사 대상 시연, URL 접근, 저장소 열람과 평가 참여는 권리 이전이나 재사용 라이선스 부여로 해석하지 않는다.
- 이 권리 주장은 `apc-monitoring-mvp`, 회사 공용 규칙, 제3자 의존성·콘텐츠·상표·자산으로 확장하지 않는다.
- 세부 적용 path, 허가와 증거 정책은 `AUTHORSHIP_AND_OWNERSHIP.md`와 `governance/intellectual-property.yaml`을 따른다.

## 제품 표면 카피

- 제품명은 `Context Console`을 작업명으로 사용한다.
- 표면에서 `MVP`, `데모`, `AI-powered`, `혁신적`, `차세대`, `사내 서비스`를 사용하지 않는다.
- 기능 이름은 `변경 요청`, `영향 분석`, `Context`, `승인`, `검증`, `활성화`처럼 실제 업무 명사와 동사를 사용한다.
- AI가 생성한 내용에는 `AI 제안`과 근거·확신도·미확정 항목을 명시하되 제품 전체를 AI로 수식하지 않는다.

## 성공 기준

- 대표 변경 요청을 5분 안에 등록→분석→검토→승인→검증→활성화 흐름으로 설명할 수 있다.
- 모든 화면의 수치·상태·행동이 동일 fixture와 상태 모델에서 파생된다.
- 승인 전 적용 불가, 실패 테스트가 있으면 완료 불가 규칙이 UI와 로직에서 일치한다.
- 기획·퍼블리싱·개발·QA 산출물의 원본 경로와 파생 관계를 확인할 수 있다.
- 정보 밀도가 충분하면서도 사용자가 다음 행동을 10초 안에 찾을 수 있다.
