## 기본 원칙

- 화면은 기능의 존재가 아니라 사용자가 내려야 할 판단과 다음 행동을 중심으로 구성한다.
- 운영 화면은 정보 밀도를 숨기지 말고 계층, 정렬, 필터, 상태, 세부 공개로 다룬다.
- AI는 검토 가능한 제안자이며 승인·정책·결정의 최종 권위처럼 표현하지 않는다.
- 장식보다 실제 데이터, 실제 상태, 실제 담당자, 실제 이력과 검증 근거를 우선한다.
- 컴포넌트 일관성은 모든 요소를 같은 모양으로 만드는 것이 아니라 같은 의미에 같은 패턴을 쓰는 것이다.

## 금지 패턴

- 보라·파랑 네온 글로우를 기본 시각 언어로 사용하지 않는다.
- 의미 없는 3D 오브젝트, 구슬, 유리 카드, 배경 광선을 사용하지 않는다.
- `AI-powered`, `Next-generation`, `Revolutionary`처럼 결과를 설명하지 못하는 추상 문구를 사용하지 않는다.
- 모든 섹션을 동일한 3개 카드 구조로 반복하지 않는다.
- 제품 맥락 없이 아이콘 수를 늘리지 않는다.
- CTA를 과도하게 크거나 번쩍이게 만들지 않는다.
- 정보가 적은데 여백만 큰 마케팅 레이아웃을 운영 화면에 적용하지 않는다.
- 실제 사용자·워크플로·운영 상태가 보이지 않는 장식용 대시보드를 만들지 않는다.
- 과도한 radius를 사용하지 않는다. 기본 radius는 6~10px, 큰 surface도 12px를 넘지 않는 것을 원칙으로 한다.
- 모든 콘텐츠를 카드로 감싸지 않는다. 독립된 경계와 배경이 필요한 그룹에만 surface를 사용한다.
- 모든 레이아웃에 border를 두르지 않는다. 표, 입력 영역, 선택 영역, 감사 증거처럼 경계가 의미를 가질 때만 사용한다.
- 모든 버튼을 색만 다른 사각형으로 만들지 않는다. primary, secondary, quiet, destructive, icon action의 계층과 affordance를 구분한다.
- 의미 없는 pill eyebrow, 장식용 badge, 기능 없는 상태 chip을 사용하지 않는다.
- 제품 표면에 `MVP`, `데모`, `프로토타입`, `실험용`, `사내 서비스` 같은 제작 단계·자기평가 표현을 노출하지 않는다.

## 운영형 UI 품질 기준

- 페이지 제목 아래에는 현재 사용자가 판단할 범위와 갱신 시점을 제공한다.
- 상태는 색상만으로 구분하지 않고 텍스트, 아이콘 또는 패턴을 함께 사용한다.
- 승인·반려·적용·배포처럼 결과가 큰 행동은 대상, 범위, 되돌리기 가능 여부를 확인시킨다.
- 표는 기본 정렬, 비어 있음, 로딩, 오류, 권한 없음, 긴 텍스트를 모두 설계한다.
- diff는 추가·삭제 색상뿐 아니라 행·필드·문서 의미를 함께 보여준다.
- 자동 검증과 수동 확인을 분리하고 누가 무엇을 확인했는지 남긴다.
- 모바일에서는 검토와 상태 확인을 우선하고, 복잡한 편집은 축약 또는 단계화한다.

## 디자인 시스템 사용

- 공통 상호작용은 검증된 디자인 시스템 컴포넌트를 우선한다.
- 디자인 시스템의 기본 테마를 그대로 브랜드로 간주하지 않는다. 의미 토큰을 통해 제품 색상과 밀도를 적용한다.
- 컴포넌트 래퍼를 만들어 제품의 상태·권한·감사 규칙을 중앙화한다.
- 접근성, 키보드 탐색, 포커스 표시를 CSS 장식보다 우선한다.
